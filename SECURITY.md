# Security Policy

Protecting the X Fresh Themes ecosystem

---

## Reporting Security Vulnerabilities

If you discover a security vulnerability in **X Fresh Themes** (including TOML configuration errors, layout exploits, or component vulnerabilities), please report it responsibly via email to:

**Email:** x@xscriptor.com

> **Do not** open public GitHub issues for security vulnerabilities. Private disclosure allows us to fix the issue before it can be exploited.

### What to Include

When reporting a security issue, please provide:

1. **Description** — A clear explanation of the vulnerability.
2. **Type** — The category of the issue (e.g., configuration injection, sensitive data exposure, etc.).
3. **Steps to Reproduce** — Detailed steps or a Proof of Concept (PoC).
4. **Impact** — How severe is the issue? What could an attacker achieve?
5. **Affected Themes** — Which specific theme or TOML configuration is affected?
6. **Proposed Fix** (optional) — Suggestions on how to patch the vulnerability.

### Guidelines

- **Confidentiality** — Do not disclose the vulnerability publicly until a fix is released.
- **Patience** — Please give the maintainers reasonable time to address the issue.
- **Response Time** — We aim to acknowledge receipt within **7 days**.

## Security Best Practices for Users

While these themes are designed to be safe, please keep these recommendations in mind:

1. **Stay updated** — Always use the latest version of the theme files.
2. **TOML Integrity** — Never include sensitive credentials (API keys, private tokens) directly in your `theme.toml` or configuration files.
3. **Sanitize Inputs** — If your Fresh setup allows user-generated content, ensure it is properly sanitized before being processed by the theme components.
4. **Trusted Sources** — Only use TOML configurations and components from the official Xscriptor repository.
